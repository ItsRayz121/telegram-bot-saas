## P4 — Revenue Analytics Wrong ✅ CONFIRMED BUG (highest confidence)

**Root cause.** MRR/ARR are **not** derived from paid subscriptions. They are a head-count
estimate that multiplies *every user whose tier string is pro/enterprise* by list price.

`backend/routes/admin.py:269-275`
```python
pro_count        = User.query.filter_by(subscription_tier="pro").count()
enterprise_count = User.query.filter_by(subscription_tier="enterprise").count()
mrr_cents = (pro_count * 900) + (enterprise_count * 4900)   # $9 / $49
arr_cents = mrr_cents * 12
```
This counts **trial users, admin-granted, promo, comp, and expired-but-not-downgraded**
accounts as paying. That is exactly why $103/$1,236 looks inflated. (Note: $103 ≠ a clean
multiple of 9/49, so the live number is already a mix — confirming it is not cash-based.)

**Files:** `backend/routes/admin.py` (`get_revenue`, lines 241-345).
**Tables:** reads `users.subscription_tier`. Should read `payment_history` (already loaded
in the same function at line 250 for other metrics) and/or a subscription table.

**Proposed fix.**
- Define MRR as the sum of **currently-active paid subscriptions**, normalized to monthly:
  count only users whose paid access is (a) backed by a `PaymentHistory` row with
  `status="confirmed"`, (b) not expired (`subscription_expires_at > now`), and (c) not a
  trial/comp/admin grant. Need to confirm which fields flag trial/comp (candidates:
  `is_trial`, `subscription_source`, `trial_ends_at` — to verify in `User` model).
- ARR = MRR × 12.
- If zero qualifying paid subs → MRR = $0, ARR = $0 (per your requirement).
- Add to the API response the **list of contributing subscriptions** (user id, tier,
  amount, last payment date) so the number is auditable in-panel.

**Verify.** Locally seed: 1 confirmed Pro payment + 1 trial Pro + 1 admin-granted Pro.
Old code → MRR $27. New code → MRR $9. Show both in the admin revenue card with the
contributing-rows list.

## P6 — Custom Bot Count Incorrect ⚠️ CONFIRMED CAUSE / NEEDS DATA for exact rows

**Root cause.** There are **two coexisting bot tables**, and different surfaces count them
differently.

| Model | Table | Used by |
|---|---|---|
| `Bot` (`backend/models.py:268`) | `bots` (legacy) | Admin **Bot Health** tab (`admin.py:800`), legacy runner `bot_manager.py` |
| `CustomBot` (`backend/models.py:1564`) | `custom_bots` (new) | Admin **Custom Bots** tab (`admin.py:712`), dashboard `/custom-bots` |

- Dashboard `list_custom_bots` (`backend/routes/custom_bots.py:30-49`) lists **`CustomBot`**
  rows and reaches into `Bot` only to patch group counts.
- Admin splits them across **two different tabs reading two different tables**.

So a user with one row in `bots` and one in `custom_bots` can legitimately show **2** in
one place and **1** in another, depending on which surface and which tab. The exact 2-vs-1
you saw depends on which table each of your real bots lives in — **NEEDS DATA**.

**Files:** `backend/routes/admin.py` (712, 800), `backend/routes/custom_bots.py` (20-51),
`backend/models.py` (268, 1564).
**Tables:** `bots`, `custom_bots`.

**Proposed fix.**
- Make admin show a **single unified bot list** that UNIONs `bots` + `custom_bots`
  (tagged with a `source` field: legacy / custom), matching how the dashboard presents
  "Community Bots". One count, one source of truth.
- Longer term: migrate legacy `bots` rows into `custom_bots` and retire the old table
  (separate, riskier change — recommend doing the union first, migration later).

**Verify (after data).** Run the diagnostic endpoint → see `bots` count, `custom_bots`
count per owner. Confirm union total == dashboard count.

## P2 — AI System Verification ⚠️ KEY FINDING: AI moderation is NOT wired into the official bot, and is key-gated/opt-in for custom bots

There are **two separate moderation code paths**:

| Path | Where | What it does | AI? |
|---|---|---|---|
| Inline `_automod_check` | `backend/official_bot.py:3497` | bad-words **substring** match + **rate-based** spam counter | ❌ No AI at all |
| `ModerationSystem.check_automod` | `backend/bot_features/moderation.py:349` | ~30 checks incl. promotional patterns, hidden-URL, **AI relevance** (`_call_ai_moderation`, Anthropic/OpenAI/Gemini) | ✅ Yes (Layer 3) |

**The official Telegizer bot only ever calls the inline rule-based path.** The full
`ModerationSystem` (the one with AI) is invoked in exactly one place:
`backend/bot_manager.py:2183` — the runner for **custom bots** (both legacy `Bot` and new
`CustomBot` route through `bot_manager.route_update`, `app.py:274`).

So, against your P2 checklist:

### Official Telegizer Bot
| Feature | Reality |
|---|---|
| AI moderation | ❌ Not wired — official bot uses rule-based `_automod_check` only |
| AI spam detection | ❌ Spam is a **rate counter** (`_spam_tracker`), not AI |
| AI link detection | ❌ Rule-based only on official bot |
| AI context moderation | ❌ Not wired into official bot |
| AI relevance checking | ❌ Not wired into official bot |
| AI anti-promotion | ❌ Official bot has no promotional-pattern check (that lives in `ModerationSystem`) |

### Custom Bots (via `bot_manager`)
AI relevance **exists and is wired** (`moderation.py:1027 check_ai_relevance` →
`_call_ai_moderation`), but only fires when **all** of these hold:
1. `automod.enabled` (default on)
2. `smart_mod.enabled` (default **off** — must be enabled)
3. `smart_mod.ai_enabled` (must be on)
4. `smart_mod.group_topic` set (non-empty)
5. message ≥ 10 words, past a 30s/user cooldown
6. **the owner has configured their own AI key** (`get_workspace_ai_key`, `moderation.py:1056`)
   — if no key, AI silently no-ops and returns `"ok"`.

**Latent bug:** `check_ai_relevance` resolves the owner via
`Bot.query.get(group.bot_id)` (`moderation.py:1050-1053`). For groups owned by a **new
`CustomBot`** (not the legacy `Bot` table), this lookup returns `None` → owner `None` →
AI relevance silently disabled. So AI moderation is effectively dead for `CustomBot`-table
bots even when configured. This is the P6 two-table split biting again.

### Telegizer Echo (assistant)
Modules exist and are wired: `assistant/handlers/_ai.py`, `personal_assistant.py`,
`hub_reply.py`, `digest_ai.py`, `embedding_service.py`, knowledge/notes/tasks handlers.
All are **AI-key-gated** the same way. Whether each returns correct results end-to-end
**requires a runtime with a valid key + live Telegram traffic** — I can't execute that
from here. I can audit each handler's wiring statically if you want a per-handler
wired/not-wired table.

**Proposed fixes.**
- Decide the product intent: should the **official bot** have AI moderation? If yes, wire
  `ModerationSystem` (or a shared engine) into `official_bot.py`'s message handler instead
  of the inline stub. If no, update the UI/marketing so it doesn't promise AI moderation
  on official-bot groups.
- Fix the `check_ai_relevance` owner lookup to resolve **both** `Bot` and `CustomBot`
  (mirror the P6 unification).
- Surface, in the admin AI audit / bot card, whether each bot owner has a working AI key
  (so "AI moderation enabled but no key" is visible, not silent).

**Verify.** Locally configure a test key, enable `smart_mod.ai_enabled` + topic, send a
promotional 12-word message through `bot_manager` → expect deletion + AI verdict logged.
Repeat against official bot → expect (currently) no AI action.

## P1 — Custom Bot Health Monitoring Is Incomplete ⚠️ PARTIAL TODAY, needs a new automated system

**What exists today.**
- Admin **Bot Health** tab (`backend/routes/admin.py:735 admin_bot_health`,
  `:844 admin_bot_health_ping`) — manual `getMe` ping + 24h error counts from
  `bot_health_events`.
- `BotHealthEvent` table (`models.py:1737`) — append-only **error log only** (no success
  pings, no last-seen liveness). Populated by `backend/health.py record_bot_error()`.
- `Bot.get_health_status()` (`models.py:303`) derives active/idle/offline from
  `is_active` + `last_active` age.
- Celery beat is live (`railway.worker.toml`, `backend/scheduler.py`) with ~25 jobs,
  including `check_group_health` (every 30 min) and `check_inactive_groups` (daily) —
  **but these monitor group *activity*, not bot *reachability***
  (`scheduler.py:743, 809`). There is **no scheduled bot-reachability check** and **no
  owner alert when a bot token dies / webhook breaks**.

**Gaps vs your spec.** No automated ping schedule; no `last_ping_at` /
`last_successful_ping` / `failure_count` columns; no owner email/Telegram alert on
failure; no escalation states (warning → critical → inactive → archived); admin shows
"reachable" not a health grade.

**Proposed system (modeled on the existing `check_group_health` job pattern).**
1. **Schema:** add to `Bot`/`CustomBot` (or a shared `bot_health_state` table):
   `last_ping_at`, `last_successful_ping`, `last_failed_ping`, `consecutive_failures`,
   `health_grade` (healthy/warning/critical/inactive/archived). Requires a `migrate.py`
   migration (release phase already runs it — `Procfile: release`).
2. **Job:** new Celery task `check_bot_health` (every 6h, configurable) that calls
   Telegram `getMe` for each active bot across **both** tables, updates the fields, and
   records failures to `bot_health_events`.
3. **Escalation:** 1 fail → warning; 3 consecutive → critical; 7 days → inactive;
   30 days → archived. Drive off `consecutive_failures` + `last_successful_ping` age.
4. **Owner alert:** on transition into warning/critical, send email (existing email
   infra used by `send_lifecycle_emails`) **and** Telegram DM (existing
   `_send_telegram_dm` used by `check_group_health`), rate-limited like the group-health
   alert (one per day per bot).
5. **Dashboard:** add a **Ping** button to each bot card in Dashboard → Community Bots
   (frontend `MyBots.js`), hitting an owner-scoped ping endpoint.
6. **Admin:** replace "reachable" with the health grade + escalation state.

**Files touched (planned):** `backend/models.py`, `backend/migrate.py`,
`backend/scheduler.py` (new task + beat entry), `backend/notifications.py`,
`backend/routes/admin.py`, `backend/routes/custom_bots.py` (owner ping),
`frontend/src/pages/MyBots.js`, `frontend/src/pages/AdminPanel.js`.

## P5 — Pending Group Investigation ⚠️ PARTIAL / NEEDS DATA

**What the code shows.** For **official-bot** groups, `bot_status` is set:
- `"active"` only when a `/linkgroup` code is redeemed (`telegram_groups.py:253`).
- `"pending"` when the bot is in the group but unlinked, or after unlink
  (`telegram_groups.py:353`).

There is **no automatic pending→active transition** — status only flips on link-code
redemption. A group where the bot is added and working but where the owner never completed
(or the system never recorded) the link step stays `pending` indefinitely. Your examples
(`CreatorX_Official_Group`, `Crypto With Fazal Discussion`) are likely **custom-bot**
groups; their status is set on a different path (`telegram_groups.py:1785` writes
`"status": "pending"`), which I need to trace fully once we confirm which model owns them
— **NEEDS DATA** (which bot/table each group belongs to, and its current `bot_status`).

**Files:** `backend/routes/telegram_groups.py` (`link_group` 250-260, `unlink_group`
352-356, pending list 404-419, 1785), `TelegramGroup` model (`models.py:1187`).
**Tables:** `telegram_groups` (`bot_status`, `owner_user_id`, `linked_bot_id`).

**Proposed fix.**
- **Auto-promote:** when a bot is confirmed admin in a group and receiving updates (we see
  messages in `message_buffer` / a successful `getChatMember` self-check), flip
  `bot_status` `pending → active` automatically — fold this into the new `check_bot_health`
  job or a dedicated reconcile pass.
- **Admin "Why Pending?"**: compute and show the concrete reason (bot not admin / missing
  permissions / never linked / no recent updates) using `getChatMember` on the bot itself
  + last-update timestamp.

**Verify (after data).** For one currently-"pending"-but-working group, run the reconcile
logic in a local script against its real chat id → confirm it flips to active and the
"Why Pending?" reason is accurate.

## P3 — Onboarding Tour Reappears ⚠️ CONFIRMED likely cause: storage, not logic

**What exists.** `frontend/src/components/OnboardingTour.js` already persists
`localStorage['onboarding_tour_v1'] = '1'` on FINISHED / SKIPPED / CLOSE, and won't run if
the key is present. A **Settings → "Retake Onboarding Tour"** button already exists
(`Settings.js:1228-1238`, calls `resetTour()`), so that requested feature is **done**.

**Why it still reappears.** The persistence is **localStorage-only** and the app runs
heavily inside the **Telegram Mini App webview** (per memory: telegizer.com/mini-app).
Telegram's in-app webview frequently does **not** persist localStorage across sessions
(ephemeral storage / different origin context), so the key is wiped and the tour shows
again on every entry. Secondary risk: if a Joyride target (`#tour-groups`, etc.) isn't in
the DOM when the tour runs (e.g. sidebar collapsed on mobile, or tour starts before
sidebar mounts), Joyride can end via `TARGET_NOT_FOUND` without firing
FINISHED/SKIPPED/CLOSE, so the key is never written. Note there is **no server-side flag**
— `User` model has no `tour_completed` column (confirmed: no match in `models.py`).

**Files:** `frontend/src/components/OnboardingTour.js`, `frontend/src/pages/Settings.js`
(reset already present), `backend/models.py` (`User` — add flag).
**Tables:** `users` (add `onboarding_tour_completed`).

**Proposed fix.**
- Make persistence **server-side**: add `users.onboarding_tour_completed`; set it via a
  small `POST /api/settings/tour-complete` on finish/skip/close; gate `run` on the
  server flag (with localStorage as a fast-path cache). Survives webview storage wipes.
- Harden the Joyride callback to also persist on `EVENTS.TOUR_END` and
  `TARGET_NOT_FOUND`, so a missing target can't loop the tour forever.
- "Reset Product Tour" then clears the server flag (button already exists; point it at the
  new endpoint).

**Verify.** Complete the tour, hard-refresh + clear localStorage → tour stays dismissed
(because server flag is set). Click Reset → tour returns once.
